# GTB 后端接口梳理 + betbus 改造可行性判定

> 依据：本仓库 `src/api/`（handlers/models/loaders）+ `.env` 服务端点实测梳理（2026-06-16）
> 用途：对照《betbus-vs-gtb-布局对比分析.md》的转型清单，判定每项改造**前端能否独立落地**，还是**必须先有后端**。

---

## 0. 结论速览

| 判定 | 含义 | 改造项 |
|------|------|--------|
| ✅ 可执行 | 现有接口/数据已够，纯前端重组即可 | 投注单悬浮化、导航专题页化、账户头像浮层、Mis Apuestas 入口、A–Z 字母筛选、移动自绘键盘 |
| ⚠️ 部分可执行 | 前端能做近似，理想形态需后端补查询 | 12h/24h 时间快筛、盘口 Tab 分类细化 |
| ❌ 需新增后端 | 无接口/无数据源，前端做不了 | 比赛详情**直播聊天室**、**天气/场馆信息头**、顶部**Transmisión 直播流**、**Bet Builder(Crear Apuesta)**、首页**中奖滚动条** |

一句话：**P0 两件大事（导航专题页化 + 投注单悬浮化）都能纯前端落地**；卡住的都集中在 betbus 的“增值内容/实时互动”能力（聊天、直播流、天气、自建注、中奖流），这些 GTB 后端目前**零支撑**。

---

## 1. 后端接口全景（按业务域）

服务端点（`.env`）：`UOF_SERVICE`（盘口）、`SPORT_SERVICE`、`USER_SERVICE`、`PAYMENT_SERVICE`、`GAME_SERVICE`（娱乐场）、`ACTIVITY_SERVICE`、`ANALYTICS_SERVICE`、`WEBSOCKET_SERVICE`（赔率推送）、`SSE_SERVICE`（余额推送）。
**无** stream/chat/transmission 专用服务。`LIVECHAT_*` 是第三方**客服**组件 license（非赛事聊天）。

| 业务域 | Handler | 主要接口 |
|--------|---------|---------|
| 体育导航/菜单 | `menu.ts` | GetTopSports、GetMenuSports、GetMenuCategories、GetMenuTournaments、GetMenuHotTournaments、GetAllHotTournaments |
| 赛事列表 | `matches.ts` | GetHotMatches、GetBestMatches、GetBestLiveMatches、SearchMatches、PostLiveMatchCount、GetBreadcrumb、GetMarketTabs、GetMarketChip、PostMatchRowBatchCount |
| 赛事详情 | `match.ts` | GetMatch、PostLocalCart、PostOutcomeNameBatch |
| 赛事/盘口 | `tournament.ts`、`champion-handicap.ts` | GetTournamentMarkets、GetOutrightMarkets(冠军盘/Futuras) |
| 投注单/下单 | `cart.ts`、`order.ts`、`parlay-boost.ts` | GetCart、PutCartItem、Get/UpdateSlipSettings、CreateOrder、GetOrderList、串关加成 |
| 战绩/交易 | `transaction-bethistory.ts`、`transaction.ts` | GetSportReport、GetGameReport、GetCasinoBetHistory、转账单 |
| 钱包/支付 | `wallet.ts`、`currency.ts`、`deposit.ts`、`withdraw.ts`、`bank-account.ts`、`recharge-code.ts`、`merchant.ts` | GetBalance 等 |
| VIP | `vip.ts` | GetUserVipInfo、GetVipLevelInfo（levelNo/levelName/levelExp/maintenanceExp）、GetVipTierInfo、GetVipActivityStatus、奖励领取 |
| 用户/认证 | `user.ts`、`passport.ts`、`user-kyc.ts` | 登录/资料/KYC |
| 娱乐场 | `casino.ts` | 大厅/游戏 |
| 统计 | `statscore.ts`、`analytics.ts` | STATSCORE 第三方比赛统计 widget |
| 通知/客服/FAQ | `notification.ts`、`support.ts`、`faq.ts`、`health-setting.ts`、`libs/livechat` | 站内信、第三方客服 |
| 促销/活动/广告 | `promotion-world-cup-league.ts`、`world-cup-pass.ts`、`recommend-card.ts`、`ad-placement.ts` | 世界杯活动、推荐卡、广告位 |

**实时通道**：`WEBSOCKET`（二进制赔率推送）+ `SSE`（余额/结算）。无聊天通道、无直播流通道。

## 2. 逐改造项可行性判定

### ✅ 可执行（纯前端，后端已够）

| 改造项 | 依赖接口/数据 | 说明 |
|--------|--------------|------|
| **投注单：固定列 → 右下角悬浮** (P0) | `cart.ts`/`order.ts`/`parlay-boost.ts`（不变） | 数据流完全不动，只改 `BetSlipPanel` 挂载位置与折叠态 UI。`sports-layout-client.tsx` 把面板从右 `aside` 拆出改 fixed 悬浮 |
| **体育导航：专题页化** (P0) | `menu.ts`（GetMenuCategories=国家 / GetMenuTournaments=联赛 / Hot=Popular）+ `matches.ts`（精选）+ `GetOutrightMarkets`（Futuras） | betbus 国家手风琴树所需数据**正是 GTB 侧栏现在用的那套**（sport→category→tournament）。改造=把树从侧栏搬进运动页正文，数据 0 改动 |
| **运动页 A–Z 字母筛选** | 现有 categories/tournaments 列表 | 纯前端按首字母分组，无需后端 |
| **Mis Apuestas 一级入口** (P1) | `transaction-bethistory.ts`（GetSportReport）/`order.ts`（GetOrderList），路由 `/account/transactions` 已存在 | 顶部/底部加入口指向现有战绩页即可 |
| **账户：头像下拉聚合浮层** (P2) | `wallet.ts`（GetBalance）+ `vip.ts`（GetUserVipInfo/GetVipLevelInfo 含 levelExp 进度）+ 奖金字段 | 余额/奖金/VIP 进度数据全有，浮层是对现有数据的前端重排；明细仍复用 `/account/*` |
| **移动投注单自绘数字键盘** (P2) | 无需后端 | 纯前端输入组件 |

### ⚠️ 部分可执行（前端可做近似，理想需后端）

| 改造项 | 现状 | 缺口 |
|--------|------|------|
| **运动页 12h/24h 时间快筛** | `match.start_time` 字段有，前端可对拉取到的赛事按时间分桶 | 无“时间窗”查询参数；要精确/全量需后端加 `start_time` 区间过滤，否则前端只能在已取列表内近似 |
| **比赛详情盘口 Tab 分类细化**（Goleadores/Córners/Tarjetas/Especiales/Combinadas） | `GetMarketTabs`/`GetMarketChip` 已按盘口返回 | 能否细分到 betbus 同款分类，取决于 UOF 盘口模板覆盖度；多数可映射，个别分类（如 Especiales）需后端配模板 |

### ❌ 需新增后端（前端无法独立落地）

| 改造项 | 为什么不行 | 需要后端提供 |
|--------|-----------|-------------|
| **比赛详情：每场直播聊天室** (P1) | 代码内 `livechat` 是**第三方客服**（license + prepareLiveChat），**非赛事公屏**；WS 通道只推赔率 | 聊天室服务：房间/消息历史 API + WS 聊天频道 + 敏感词/封禁 + VIP 徽章 |
| **比赛详情：天气/场馆信息头** (P1) | `match.ts` 模型只有比分/钟表/统计（黄牌/角球等），**无 weather/venue/stadium/referee/attendance** | 在赛事详情接口补天气、体育场、容量、裁判等字段（需上游数据源） |
| **顶部 Transmisión 直播流** (P1) | 全仓库**无任何 stream/transmission 代码或服务端点**，赛事模型无 `stream_url` | 流媒体服务 + 赛事流地址 + 版权/分发 |
| **Bet Builder（Crear Apuesta 自建投注）** (P1) | 无 SGP/bet-builder 接口；`order/cart` 是常规单注/串关 | 同场关联盘（SGP）定价与下单接口 |
| **首页 Ganancias Recientes 中奖滚动条** (P2) | 无任何中奖/赢家 feed 接口 | 近实时中奖流 API（或 WS 频道） |

## 3. 落地建议

- **可立刻排期（前端为主）**：投注单悬浮化、导航专题页化、A–Z 筛选、Mis Apuestas 入口、账户头像浮层、移动键盘。这批覆盖了用户最在意的两处结构性差异（P0），且不阻塞于后端。
- **需先提后端需求**：聊天室、天气场馆头、直播流、Bet Builder、中奖滚动条。建议把这 5 项整理成后端接口需求单，排期前先确认数据源/上游版权（直播流、天气尤其受限于第三方）。
- **建议先做布局骨架、内容降级**：聊天室/直播/中奖这类，前端可先留位（占位组件 + feature flag 关闭），后端就绪再点亮，避免阻塞整体转型。

> 判定基于现有代码静态梳理；个别接口是否支持额外参数（如时间窗、盘口模板），建议与后端二次确认。
