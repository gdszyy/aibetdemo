# 文件截断 / 损坏问题 · 根因分析

> 适用范围：本次「投后周报技能改造」会话中，文件被截断 / 内容损坏的现象与成因。
> 一句话结论：**根因是这个沙箱环境的"文件系统挂载层"在读取/复制时返回了不完整数据，不是 Edit 工具、也不是脚本编码 bug。**

---

## 1. 结论速览（TL;DR）

- 现象：部分文件读出来 / 复制出来后**被截断或字节损坏**（行数变少、JSON/Python 解析失败、UTF‑8 在中途断裂），但**字节大小有时还和源文件一样**。
- 真凶：**两套文件系统视图不一致 + 挂载层"短读（partial read）"**。文件工具（Read/Write/Edit）走的是宿主 Windows 原生磁盘；Bash 在独立 Linux 沙箱里，通过一层网络/虚拟挂载访问同样的目录。跨层读取是惰性/带缓存的，**单次读可能只返回一部分字节**。
- 谁不背锅：**Edit 工具不背锅**——它的写入在 Windows 原生视图里是完整的；只有当这份内容随后被 Bash 挂载层"读出来 / cp / 打包"时才被截短。
- 可靠的做法：**直接写**（heredoc、`python open().write()`）+ **用文件工具读校验**（Windows 原生视图是权威）。
- 不可靠的做法：**`cp` / 挂载读大文件 / `zip` 打包**——都可能截断。

## 3. 不是什么（排除项）

- **不是 Edit 工具截断**：Edit 的写入在 Windows 原生视图（文件工具复读）里是**完整**的。截断只在内容**经 Bash 挂载层再次读出/复制/打包**时出现。换句话说：Edit 负责"写对"，挂载层负责"读漏"。
- **不是脚本编码 bug**：`load_registry()` 等都显式 `encoding="utf-8"`；坏掉的是**字节本身不完整**，不是解码方式。
- **不是我手抖写漏**：直接 heredoc 写出的文件经文件工具复核均完整；坏的都是"复制/挂载读"得到的副本。
- **不是简单的"文件变短"**：现象 1–3 的字节大小与源**相同**，却仍解析失败——说明存在**字节替换 / 中途短读拼接**，不只是末尾截断。

---

## 4. 根本机制：两套文件系统视图不一致

```
文件工具 (Read/Write/Edit) ──► 宿主 Windows NTFS（原生、权威、可靠）
                                     ▲
                                     │ 经一层网络/虚拟挂载（9p / virtiofs / FUSE 类）暴露
                                     ▼
Bash 沙箱 (cp / cat / zip / python open) ──► 挂载视图（带缓存、可能短读/惰性同步）
```

- **写方向**：直接写（heredoc / `open().write()`）能完整落到 NTFS——所以文件工具复读是完整的。
- **读方向**：挂载层对**较大文件**容易发生 **partial read（短读）**——一次 `read()` 返回的字节数少于文件长度。不"循环读到 EOF"或对短读不重试的工具（`cp`、`zip`、甚至某些 `python` 读路径）就拿到**截断**数据。
- **跨层缓存不一致**：文件工具写到 NTFS 后，Bash 挂载视图可能仍是**旧/部分**内容（现象 7：Edit 写对了、Bash 看到旧的截断版）。

这套模型同时解释了所有现象，包括最反直觉的「`ast.parse` 说 OK，但末行是半截」——**同一进程里不同的 `read()` 调用返回了不同的字节量**（一次读到全、一次短读），正是 partial read 的典型表现。

## 5. 候选原因逐条分析

| 候选原因 | 证据 | 判定 |
|---|---|---|
| **A. 挂载层短读（partial read）** | 现象 6 的"OK 与截断并存"；大小相同却内容不全；大文件高发 | ✅ **主因** |
| **B. 多字节字符在短读边界被切半** | `project-lark-index.json` 末尾 UTF‑8 断裂；`data-registry.json` 切在引号上→合法 UTF‑8 但 JSON 坏 | ✅ 主因的直接后果 |

| **D. `cp` / `zip` 信任单次读、不校验** | outputs 内部 `cp` 把 `fetch_all_data.py` 截断；read‑only 技能挂载 `cp -r` 损坏多文件 | ✅ 放大器 |

| G. 区域/编码（locale）解码错 | 脚本均显式 utf‑8；坏的是字节不全 | ❌ 排除 |
| H. Edit 工具本身写截断 | 文件工具复读 Edit 产物完整 | ❌ 排除 |
| I. 我重建时手误写漏 | 直接写的文件复核完整 | ❌ 排除 |

## 6. 为什么"大文件 / 多字节文件"更容易中招

- **短读概率随文件增大而上升**：小文件常能一次读完；大文件需要多次 `read()`，任意一次短读就可能被不健壮的工具当成 EOF。
- **中文/多字节放大破坏面**：短读边界落在多字节字符中间 → 末尾非法 UTF‑8（现象 1）；落在引号/结构符上 → 字节合法但 JSON/代码语法破裂（现象 2、3）。这也是为什么"大小一样却解析失败"。
- **经验阈值**：本次 6 KB 以上的文件均出现过损坏；43 KB 的 `report-skeleton.html` 属高危（正因如此没有把它重新打包）。

## 7. 可靠 vs 不可靠操作对照

| 操作 | 是否可靠 | 说明 |
|---|---|---|
| 文件工具 `Read` | ✅ 权威 | 走 NTFS 原生，作为"内容真值"校验基准 |
| 文件工具 `Write` | ✅ | 直接落 NTFS；写完用 `Read` 复核 |
| Bash heredoc 写 (`cat > f <<'EOF'`) | ✅ | 直接写，内容来自进程内、完整落盘 |
| `python open().write()` | ✅ | 同上 |
| Bash 读挂载里的**大/多字节**文件 (`cat`/`open`/`grep`) | ⚠️ 不稳定 | 可能短读；解析类检查可作辅助但非权威 |
| `cp` / `cp -r` | ❌ | 信任单次读，会截断 |
| `zip` / 打包整目录 | ❌ | 逐文件读=同样的短读风险（故未产出单个 `.skill`）|
| Bash 删除 outputs 文件 (`rm`) | ❌ | 本次 `rm` 一律失败（Operation not permitted）|
| `/tmp` 写执行 | ❌ | 本次 `/tmp` 报 Permission denied，改用 outputs 子目录 |

## 8. 防御与检测手册（可操作）

**写入侧**
- 重要文件**直接写**（heredoc / `open().write()`），不要用 `cp` 搬运。
- 内容的"真值"以**文件工具 Read** 为准；Bash 读到的异常优先怀疑挂载层而非内容本身。

**复制侧（必须搬运时）**
- 用"读+写+再读断言"的带校验复制，而不是 `cp`：

  ```python
  s = open(src, encoding="utf-8").read()
  open(dst, "w", encoding="utf-8").write(s)
  assert open(dst, encoding="utf-8").read() == s   # 不等就重试
  ```
- 对会"短读"的源，循环读到长度达标再用；不达标就重试若干次。

**校验侧（多重，缺一不可）**
- **结构解析**：JSON 跑 `json.load`、Python 跑 `compile`/`ast.parse`。
- **尾部校验**：检查最后一行是否为期望结尾（截断总是丢尾巴）。
- **长度/行数**：与已知源比对（但**不能只看字节大小**——现象 1–3 大小相同却坏了）。
- **权威复核**：关键文件用**文件工具 Read 尾部**确认（NTFS 视图）。
- 注意：**"UTF‑8 合法"不等于"内容完整"**（`data-registry.json` 反例）。

**打包侧**
- 在这种环境里**避免**直接 `zip` 整目录交付 `.skill`；改为逐文件"带校验读入内存→写归档→再解包逐文件解析"，任一文件校验失败就重试。
- 不确定能保证时，**宁可交付逐个校验过的单文件**，也不要交付可能内含损坏文件的压缩包。

## 10. 给后续交付的建议

- **改动文件单独交付**（本次采用）：4 新 + 3 改逐个用文件工具复核结尾，覆盖进已安装技能；不重打包未改动的大文件（如 43 KB 的 `report-skeleton.html`），避免把本来完好的文件打坏。
- 若**必须**产出单个 `.skill`：在更稳定的本地环境打包，或按 §8「打包侧」做"逐文件带校验入归档 + 解包复验 + 失败重试"。
